## 文档系统
主要是想折腾一个开发之前帮助自己理清思路，而且集成项目文档，接口文档为一体的文档系统，具体的我还没理清思路，先把看书中有关的内容罗列出来让自己慢慢思考

```

1.严格遵守正交性(改动A，绝对不会影响B)让你的代码保持解耦，避免用全局变量，避免写相似的函数（参考<<设计模式>>策略模式）。修改BUG的时候也是评估整个系统正交性的好时候，当你遇到问题时，评估修正的局部化成都。运用正交性原则，可以降低各组件间的相互依赖。（这点我觉得尤其适合你开发中期的时候重新审视你的项目代码和组件依赖。）

2.原型制作是一种学习经验，其价值并不在于所产生的代码，而在于所学到的经验教训。适合制作原型的有：架构，已有系统中的新功能，外部数据结构或内容，第三方工具或组件。性能问题。用户界面设计。


3.架构原型中解答具体问题:1.主要组件的责任是否得到了良好定义2.主要组件的协作是否得到良好定义3.耦合是否最小化4.你能否确定重复的潜在来源5.接口定义和各项约束是否可以接受6.每个模块在执行过程中是否能访问其所需要的数据，能否在需要的时候进行访问（这段让我思考到我们设计分析需求的时候往往是凭借个人经验去组合模块，但是一些潜在的比如约束等等都不大会想到，当然这些我也是在软件工程这门课备考的时候所看到的，这对于开发的确是起到引领性的作用。）


4.反馈流程：1.源代码是符合语法的2.软件的单元测试必须是成功的3.软件应该满足一定的质量标准（测试覆盖率，其它度量标准）4.软件的功能验收测试必须是成功的（检查应用是否满足业务验收条件）5.软件的非功能测试必须是成功的（检验程序是否满足用户对性能，有效性，安全性等方面的要求）6.软件必须通过探索性测试（即给客户演示功能）


5.构建可执行代码并对其测试应该是自动化的，每次提交都对应用程序进行构建并测试。


6.软件交付的原则：1.为软件的发布创建一个可重复且可靠的过程2.将几乎所有事情都可自动化3.把所有的东西都纳入版本控制4.提前并频繁做让你感到痛苦的事情。（这点我觉得开发中倒是挺有效的，就是先处理困难的，然后越到后期越轻松~，如果写测试是让你比较痛苦的，那么一开始就写测试是最好的选择，如果文档是难点，没写完一个功能就应该完善文档）5.内建质量（越早发现缺陷，修复它们的成本越低。）


7.为以下内容制定一个保存基线和控制变更的策略：1.源代码，构建脚本，测试，文档，需求，数据库脚本，代码库，配置文件2.用于开发，测试和运维的工具集3.用于开发，测试和生产运行的所有环境


8.验收测试中最好在测试与用户界面之间增加一个抽象层，以减少界面变更导致的工作量，或者通过公共API来运行验收测试。（要求UI层不含逻辑）


9.实现一个部署流水线：1.对价值流建模并创建一个可工作的简单框架（画出提交到发布整个过程的价值流图，然后持续集成和发布管理工具对流程建模。等项目全部展开，把已有的构建，测试和部署脚本放进去。这里可以先开发一个迭代0：在提交阶段还没写代码和单元测试时，写一个最简单的”hello word”再写一个单元测试，这个测试只测试”assert(true)”，然后完成部署，在网页上打开进行验收测试。）2.将构建和部署流程自动化3.将单元测试和代码分析自动化4.将验收测试自动化5.将发布自动化 部署流水线应该记录每个流程的开始和结束时间




```

现实中开发文档涵盖以下内容：

```
1.设计说明。包括架构、技术选型、环境限制、关键算法
2.外部API文档，对外所有的参数、返回值和行为都文档化。（这里我认为可以是写测试的时候把这部分完善，然后红灯，绿灯。）
内部函数不文档，不过我认为注释可以代替
3.进度控制

```
